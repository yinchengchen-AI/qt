## 1. 手册总览

### 1.1 系统定位

本系统用于支撑 **杭州企泰安全科技** 主业务链路的全流程管理:
- **业务侧**:客户线索 → 合同签约 → 项目交付
- **财务侧**:开票申请 → 回款登记 → 对账核销
- **管理侧**:业绩统计 / 账龄分析 / 公告与消息 / 数据字典 / 部门与角色

### 1.2 核心模块

| 模块 | 关键对象 | 状态数量 | 关键节点 |
|---|---|---|---|
| 客户 | 客户主数据 + 联系人 + 跟进记录 | **无状态机** | v0.5.0 整体下线（详见 §15） |
| 合同 | 合同主数据 + 附件 + 自动化记录 | 3 | DRAFT → ACTIVE → CLOSED |
| 项目 | 已下线（v0.3.0） | — | — |
| 开票 | 发票主数据 + 附件 + 审核记录 | 6 | DRAFT → PENDING_FINANCE → ISSUED |
| 回款 | 回款主数据 + 核销记录 | 5 | PLANNED → CONFIRMED → RECONCILED |

### 1.3 阅读建议

- **业务人员 / 业务人员**:重点读 4-10 章
- **财务人员**:重点读 8、9、11 章
- **行政 / 管理员**:重点读 3、12 章
- **新用户**:先看 2(快速开始)+ 4(工作台),再按需查阅

## 2. 快速开始

### 2.1 访问系统

- **开发环境**:<http://localhost:3000>
- **测试 / 预发 / 生产环境**:由管理员提供地址
- **推荐浏览器**:Chrome / Edge / Safari 最新版;移动端使用手机自带浏览器

### 2.2 登录

1. 打开登录页,左侧为 **企泰安全** 品牌区,右侧为登录表单
2. 输入 **工号** + **密码**(均由管理员分配,**无默认密码**;`create-admin` 强制 ≥ 8 字符)
3. **「7 天内自动登录」** 复选框:勾选后浏览器保留登录态 7 天;不勾选则按当前会话有效,关闭浏览器即登出
4. 点击「登录」

> **工号大小写不敏感**:`Admin` 与 `admin` 视为同一账号;登录时统一小写, 与 `prisma/seed.ts` 一致.

> **首次登录会被强制改密**(仅当管理员或迁移脚本为账号标记了 `mustChangePassword`): 登录成功后跳到 `?resetToken=...` 改密页, 新密码由用户自己设定; 改密成功后跳回登录.

#### 忘记密码(自服务重置,v0.10.0 起)

登录页右下 **忘记密码** → 弹窗输入 **工号 + 预留邮箱** → 提交. 系统会校验匹配并生成 30 分钟有效的重置链接, 由管理员通过内部渠道(电话 / 现场 / 后续 IM)送达; 拿到链接后浏览器打开 `https://<host>/login?resetToken=xxx`, 设置新密码(≥ 8 字符)即可登录.

> 重置链接是**一次性**的, 用过即失效; 同时存在重置与失效两个旧链接时以最新为准. 30 分钟未使用自动失效, 需要重新申请.

#### 失败计数与锁定

连续 5 次密码错误 → 账号被锁 15 分钟; 第 6 次起被锁 60 分钟; 距上次失败 30 分钟后再尝试视为新一轮. 锁定期间即使密码正确也无法登录. 管理员可在「用户管理 → 重置密码」清掉锁定并下发新密码.

### 2.3 切换工作区

- 顶部 **角色 / 部门** 信息会随登录用户变化
- 左侧导航 **可折叠**(桌面端),点击汉堡按钮收起为图标条
- **面包屑** 在顶部居左,可点击跳转到上级页面

### 2.4 退出

- 点击右上角 **头像 → 退出登录**
- 闲置 8 小时(不勾选自动登录)或 7 天(勾选)后系统自动登出

## 3. 角色与权限

系统内置 5 种角色,对应不同工作场景。权限边界在 **客户端 + 服务端 双重校验**,前端看不到的菜单 / 按钮同时后端也会拒绝。

> 角色定义在 `prisma/seed.ts`(`seedRoles()`)和 `scripts/shared/seed-roles.ts` 中,以 `ADMIN / SALES / FINANCE / OPS / EXPERT` 为准。本节不再列举具体账号 — 账号由管理员通过 **系统管理 / 用户管理** 创建并分配角色,初始管理员用 `pnpm create-admin` 创建。

### 3.1 角色一览

| 角色 | 典型工作 |
|---|---|
| `ADMIN`(管理员) | 全权限,负责系统配置 / 角色 / 字典 |
| `SALES`(业务人员) | 客户 / 合同 / 项目推进,**行级隔离**只看自己的数据 |
| `FINANCE`(财务人员) | 发票 / 回款 / 对账,统计可导出 |
| `OPS`(行政人员) | 部门 / 公告 / 字典维护,基础信息录入 |
| `EXPERT`(技术专家) | 任务执行 / 跟进 / 现场交付,服务进度推进 |

### 3.2 模块权限矩阵(简版)

| 模块 \ 角色 | ADMIN | SALES | FINANCE | OPS |
|---|---|---|---|---|
| 客户 | CRUD + 导出 | CRU + 导出 + **行级隔离** | R + 导出 | CRU + 导出 |
| 合同 | CRUD + 导出 | CRU + 导出 | R + 导出 | R + 导出 |
| 项目 | CRUD + 导出 | CRU + 导出 | R + 导出 | CRU + 导出 |
| 开票 | CRUD + 导出 | CR + 导出 | CRUD + 导出 | R + 导出 |
| 回款 | CRUD + 导出 | CR + 导出 | CRUD + 导出 | R + 导出 |
| 统计分析 | R + 导出 | R | R + 导出 | R |
| 部门 / 用户 / 角色 | CRUD(角色只读) | R | R | 部门 CRUD;用户 R |
| 数据字典 | CRUD | R | R | R |
| 公告 | CRUD | R | R | CRUD |
| 操作日志 | R | — | — | — |
| 消息中心 | CRUD | CRUD | CRUD | CRUD |

> **C**=Create 新建, **R**=Read 查看, **U**=Update 编辑, **D**=Delete 删除。

### 3.3 关键权限设计

- **SALES 行级隔离**:业务人员只能看到自己作为 **「客户负责人」(`ownerUserId`)** 的客户 / 合同 / 项目 / 发票 / 回款。试图越权访问他人数据,接口返回 `404`。
- **不允许越权编辑**:合同 / 发票的状态机推进由 **服务层事务** 守护,前端按钮隐藏 + 后端校验双重保险。
- **数据字典 / 角色 / 部门** 等系统配置只有管理员(或指定角色)可改;其他角色只读。
- **30 秒缓存**:用户角色 / 状态有 30 秒缓存,管理员调整后最迟 30s 在其他端生效。

## 5. 客户管理

**路径**:`客户管理 / 客户列表`
**入口**:左侧导航 → 「客户」

### 5.1 客户状态(v0.5.0 起下线)

v0.5.0(2026-06-29)起客户表不再有 `status` 字段, 详情页右上角「变更状态」入口已移除。客户跟进(`FollowUp`)在 v0.3.1 已软下线。

> 业务方后续定义新流程。完整下线原因 / 历史约束见
> [docs/superpowers/specs/2026-06-29-customer-status-deprecation.md](../superpowers/specs/2026-06-29-customer-status-deprecation.md)。
>
> 软删除的客户在列表里默认隐藏;管理员可在筛选中开启「含已删除」。

### 5.2 列表功能
### 5.2 列表功能

- **筛选**:按状态、类型、规模、行业、来源渠道、负责人、关键字
- **导出**:右上角「导出」按钮,按当前筛选条件导出为 **Excel(.xlsx)**
- **批量**:暂未开启批量操作,需逐条处理
- **列**:
  - 客户编号(`QT-C-YYYYMM-####`,系统自动)
  - 客户全称 / 简称
  - 客户类型 / 行业 / 规模(白名单字典)
  - 负责人 + 归属部门
  - 当前状态
  - 创建时间
  - 操作:查看 / 编辑 / 软删

### 5.3 新建客户

**路径**:`客户列表 → 右上「新建客户」`

表单分 4 个分区:

1. **客户基础信息**:全称 *、简称、统一社会信用代码(企业必填,18 位,自动 GB 32100-2015 加权校验)
2. **类型与规模**:客户类型 *(企业/政府/其他)、客户规模、行业
3. **所在地与来源**:省 / 市 / 区、来源渠道(展会/转介绍/官网等)、详细地址
4. **联系人**:主联系人姓名 / 职务 / 电话 *(必填,作为后续跟进主联系入口)

提交后:
- 系统自动生成 **客户编号**(`QT-C-YYYYMM-####`,并发安全,行锁)
- 创建人 / 创建时间 由系统写入
- 跳转到客户详情页

### 5.4 客户详情

详情页包含 4 个区域:

- **基础信息**(只读字段 + 编辑入口)
- **联系人列表**:支持多联系人,「主联系人」标识,可在客户详情内新增 / 编辑
- **跟进记录**:按时间倒序展示 **跟进方式 / 内容 / 跟进结果 / 下次跟进时间**;支持「新增跟进」抽屉
- **关联合同 / 项目 / 发票 / 回款**:以 ProTable 内嵌展示,带横向滚动,移动端可滑

### 5.5 客户编辑

- 仅「客户全称 / 简称 / 信用代码 / 联系人」等基础信息可改
- 状态变更在 **详情页右上角** 走专用按钮,触发跨模块校验
- 跟进记录一旦提交不可修改,只能新增(留痕)

## 6. 合同管理

**路径**:`合同管理 / 合同列表`

### 6.1 合同状态机(3 态 + 自动化)

```
        ┌──────────────┐
        │  DRAFT 草稿  │
        └──────┬───────┘
               │ [auto] 字段完整 + 至少 1 附件
               │ [admin] 强制发布
               ▼
        ┌──────────────┐
        │ ACTIVE 生效中│
        └──────┬───────┘
               │ [auto] 开票足额 (≥ 总额 × 95%)
               │ [auto] endDate < now
               │ [admin] 强制完结 (选 reason)
               ▼
        ┌──────────────┐
        │ CLOSED 已完结│
        └──────────────┘
```

| 状态 | 含义 | 可操作 (admin 视角) |
|---|---|---|
| `DRAFT` 草稿 | 字段/附件未就位 | 强制发布 / 编辑 / 删除 |
| `ACTIVE` 生效中 | 业务正常推进,开票/回款 | 强制完结(选 reason) / 编辑 / 删除 |
| `CLOSED` 已完结 | 终态 | 编辑(修正字段) / 删除(子数据兜底) |

**自动化细节**:
- `DRAFT → ACTIVE`:`createContract` / `updateContract` 保存时自动判定 `isPublishable(c)`,条件满足即升 ACTIVE。每小时 cron `tickPublishableDraffts` 兜底。
- `ACTIVE → CLOSED (completed)`:每日 `tickCompletionCandidates` 扫开票足额合同,自动完结。
- `ACTIVE → CLOSED (expired)`:每日 `runContractExpiryJob` 扫 `endDate<now` 的合同,自动完结。
- `ACTIVE → CLOSED (terminated)` / `completed`:admin 强制完结弹窗选 reason。
- 业务自创建/维护合同,日常无需手动点状态按钮。

**时间线**:详情页"自动化记录"区会拉 `ContractReviewLog`,把 `AUTO_PUBLISH` / `AUTO_CLOSE_COMPLETED` / `AUTO_CLOSE_EXPIRED` / `MANUAL_PUBLISH` / `MANUAL_CLOSE` 条目突出显示。

### 6.2 关键校验(系统自动)

- **R-04**:`→ ACTIVE` 必须字段完整(`signDate/startDate/endDate/totalAmount/taxRate/ownerUserId/signerId`) + 至少 1 个 **PDF / Word / Excel / 图片** 附件,否则保持 `DRAFT`
- **R-09**(在发票模块):开票抬头 / 税号 必须与合同 / 客户档案一致
- **R-08**:同一合同累计开票 ≤ 合同总额(具体见开票章节)

### 6.3 新建合同

**路径**:`合同列表 → 右上「新建合同」`

核心字段:

| 字段 | 必填 | 说明 |
|---|---|---|
| 客户 * | ✓ | 选 `NEGOTIATING` / `SIGNED` 客户 |
| 合同标题 * | ✓ | 简短业务描述 |
| 服务类型 * | ✓ | 字典 `SERVICE_TYPE`:安全咨询 / 安全培训 / 隐患排查 / 应急预案 / 安全评价 / 其他 |
| 签约日期 * | ✓ | 默认今天 |
| 服务期 * | ✓ | 起始日 + 截止日,截止日 > 起始日 |
| 合同总额(含税) * | ✓ | 数字,≥ 0 |
| 税率 * | ✓ | 默认 6%,可改;`税额` / `不含税金额` 自动算 |
| 付款方式 * | ✓ | 一次性 / 按阶段 / 按月 / 按季度 |
| 分期计划 | 条件必填 | 选了「按阶段 / 按月 / 按季度」时录入 JSON 计划 |
| 附件 | — | 选填;**强烈建议在提交审批前上传** |

提交保存为 `DRAFT`,可继续编辑。

### 6.4 附件上传

合同详情 / 编辑页有 **「附件」** 卡片:

- **白名单 MIME**:`application/pdf`、`application/msword`、`application/vnd.openxmlformats-officedocument.wordprocessingml.document`、`application/vnd.ms-excel`、`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`、`image/jpeg`、`image/png`、`image/webp`
- **单文件 ≤ 20MB**;**单合同附件 ≤ 5 个**
- 点击上传 → 后端签发 **5 分钟有效的 PUT URL** → 浏览器 **直传 MinIO**,不经业务服务器中转
- 上传完成的文件在卡片中显示为可点击链接,点击 **预览 / 下载**(重新签发 5min GET URL)
- **删除**只软删 `Attachment` 表记录,MinIO 对象保留(后续由 GC job 回收)

### 6.5 状态自动流转

新模型下,合同状态机 **不再有审批环节**。业务自创建/维护,admin 兜底;自动化处理 `DRAFT → ACTIVE` 和 `ACTIVE → CLOSED`。

详见 [§6.1](#61-合同状态机3-态--自动化) 状态机图和触发条件表。

`ContractReviewLog` 仍然记录所有状态迁移(AUTO_PUBLISH / AUTO_CLOSE_COMPLETED / AUTO_CLOSE_EXPIRED / MANUAL_PUBLISH / MANUAL_CLOSE),详情页时间线可见,作为审计追溯。

## 7. 项目管理（v0.3.0 已下线）

> **本节保留为历史参考**。项目管理模块（含 Project 表、7 态状态机、ProjectProgressLog、里程碑、PaymentAllocation）已于 **v0.3.0 (2026-06-23)** 下线。当前系统流：客户 → 合同 → 开票 → 回款。

**原路径**:`项目管理 / 项目列表`（已移除）

### 7.1 项目状态机(7 态，已下线)

```
PLANNED ─start─▶ IN_PROGRESS ─deliver─▶ DELIVERED ─accept─▶ ACCEPTED ─close─▶ CLOSED
   │                  │                       │                  │
   │                  │                       │                  └─cancel▶ CANCELLED
   │                  └─suspend / resume
   │
   └─cancel▶ CANCELLED
```

| 状态 | 含义 |
|---|---|
| `PLANNED` 计划中 | 合同已生效,项目尚未启动 |
| `IN_PROGRESS` 进行中 | 已启动,正在执行 |
| `(已合并入 ACTIVE)` | 暂停(等客户反馈 / 付款等) |
| `DELIVERED` 已交付 | 服务已交付,待客户验收 |
| `ACCEPTED` 已验收 | 客户已签收 |
| `CLOSED` 已关闭 | 流程结束,可开票 / 回款 |
| `CANCELLED` 已取消 | 终止 |

### 7.2 关键校验

- **R-05**:项目必须挂在一份 `ACTIVE` / `ACTIVE` 合同下
- **R-06**:**项目结束日期 ≤ 合同结束日期**,否则拒绝 `PROJECT_DATE_OUT_OF_RANGE`
- **合同状态变更时**:关联项目的状态会被联动校验(合同 COMPLETED → 所有项目必须 CLOSED)

### 7.3 新建项目

| 字段 | 必填 | 说明 |
|---|---|---|
| 合同 * | ✓ | 选 `ACTIVE` / `ACTIVE` 合同 |
| 项目名称 * | ✓ | 同一合同下唯一 |
| 服务范围 * | ✓ | 长文本 |
| 项目经理 * | ✓ | 选本系统用户 |
| 服务期 * | ✓ | 必须在合同期内 |
| 预算金额 | — | 内部预算,可选 |
| 里程碑计划 | — | JSON,描述阶段交付物 |

### 7.4 进度日志

- 项目详情页「进度日志」Tab

## 8. 开票管理

**路径**:`开票管理 / 开票列表`
**核心角色**:FINANCE 主导,SALES 发起申请

### 8.1 发票状态机(6 态)

```
DRAFT ─submit─▶ PENDING_FINANCE ─issue─▶ ISSUED
                       │                      │
                       │ reject               │ void
                       ▼                      ▼
                   (回 DRAFT)              VOIDED
                       │
                       │ redFlush
                       ▼
                  RED_FLUSHED (红冲,关联原票)
```

| 状态 | 含义 |
|---|---|
| `DRAFT` 草稿 | SALES 录入,待提交 |
| `PENDING_FINANCE` 待财务审核 | 财务介入 |
| `ISSUED` 已开 | 财务已开具,正式号已分配 |
| `REJECTED` 已驳回 | 财务驳回,回到 DRAFT |
| `VOIDED` 已作废 | 已开但作废,需关联红冲 |
| `RED_FLUSHED` 红冲 | 负数发票,关联原 ISSUED 票 |

### 8.2 关键校验

- **R-08**:**同一合同累计开票金额 ≤ 合同总额**,超额时拒绝 `INVOICE_OVER_LIMIT`
- **R-09**:`→ ISSUED` 必须有 **抬头 / 税号 / 开户行 / 账号 / 地址电话** 等发票信息
- 抬头 / 税号 通常自动取自 **客户档案**;如有差异可手工调整
- **草稿编号**:`DRAFT-{timestamp}`,正式号在 `issue` 时由系统重排

### 8.3 新建开票

| 字段 | 必填 | 说明 |
|---|---|---|
| 合同 * | ✓ | 仅 `ACTIVE` / `ACTIVE` 合同 |
| 客户 * | ✓ | 自动取合同客户 |
| 发票类型 * | ✓ | 增值税专用 / 增值税普通 / 增值税电子 / 电子普通 |
| 金额(含税) * | ✓ | 不超过合同剩余可开票额 |
| 税率 * | ✓ | 默认 6% |
| 申请日期 * | ✓ | 默认今天 |
| 预计开票日期 | — | 用于提醒 |
| 抬头类型 * | ✓ | 公司 / 个人 |
| 抬头名称 * | ✓ | 自动从客户档案带出 |
| 税号 | — | 公司抬头时建议填写 |
| 开户行 / 账号 / 地址 / 电话 | — | 用于专票 |
| 附件 | — | 开票后回传的发票扫描件 / 电子发票 PDF |

### 8.4 提交 / 审核流程

- **提交**:`DRAFT → PENDING_FINANCE`,触发领域事件 `CONTRACT_PENDING_REVIEW` 同类通知(管理员 + 财务)
- **审核通过**(财务):`→ ISSUED`,系统分配正式号
- **驳回**:`→ DRAFT`,必填意见,申请人收到 `INVOICE_REJECTED` 通知
- **作废**:`ISSUED → VOIDED`,必填作废原因
- **红冲**:`ISSUED → RED_FLUSHED`,生成反向负数发票,自动关联原票

## 9. 回款管理

**路径**:`回款管理 / 回款列表`
**核心角色**:FINANCE 录入 + 核销,SALES 申请

### 9.1 回款状态机(5 态)

```
PLANNED ─confirm─▶ CONFIRMED ─reconcile─▶ RECONCILED
     │                │
     │                ├─refund─▶ REFUNDED
     │                └─cancel─▶ CANCELLED
     │
     └─cancel▶ CANCELLED
```

| 状态 | 含义 |
|---|---|
| `PLANNED` 已计划 | 客户已通知打款,银行未到账 |
| `CONFIRMED` 已确认 | 财务已确认到账 |
| `RECONCILED` 已核销 | 已分配到具体发票 |
| `REFUNDED` 已退款 | 误收 / 重收 退款 |
| `CANCELLED` 已取消 | 误录 / 撤回 |

### 9.2 关键校验

- **R-10**:`bankRefNo`(银行流水号)在 `CONFIRMED` 状态全局唯一,重复 confirm 会被拒 `PAYMENT_DUPLICATE_REF`;**`PLANNED` 阶段允许重复**(草稿期间填错不阻塞)
- **R-11**:回款到单张发票的累计金额 ≤ 该发票金额
- **R-12**:回款到单张合同的累计金额 ≤ 该合同总额

### 9.3 新建回款

| 字段 | 必填 | 说明 |
|---|---|---|
| 客户 * | ✓ | 自动取合同客户 |
| 合同 * | ✓ | 选 `ACTIVE` / `ACTIVE` 合同 |
| 发票 | — | 若已开票可挂接 |
| 金额 * | ✓ | ≤ 合同剩余未回款额 |
| 到账日期 * | ✓ | 默认今天 |
| 收款方式 * | ✓ | 银行转账 / 支票 / 现金 / 微信 / 支付宝 / 其他 |
| 银行流水号 | — | `PLANNED` 阶段允许重复,confirm 前补齐 |
| 开户行 | — | 建议填写 |
| 备注 | — | 自由文本 |

### 9.4 确认 / 核销

- **确认到账**:`PLANNED → CONFIRMED`,流水号此时起全局唯一
- **核销**:`CONFIRMED → RECONCILED`,由 finance 在「对账」动作里勾选目标发票即可
- 一笔回款对应 **一张发票**(`payment.invoiceId`),没有跨发票分摊 —— 回款在创建时已绑定发票,避免一笔回款挂多张发票带来的跨合同抹账风险
- **退款**:`→ REFUNDED`,必填退款原因
- **取消**:`PLANNED / CONFIRMED → CANCELLED`,`RECONCILED` 状态不能直接取消(需先反核销)

## 11. 统计分析

**路径**:`统计分析 / 总览 / 账龄 / 业绩`

### 11.1 总览(Overview)

- 时间区间选择(默认本月)
- **KPI 卡片**:客户总数(含区间内新增数)、合同金额、开票金额、回款金额
- **时间序列图**:按日 / 周 / 月的合同 / 开票 / 回款趋势(用 `@ant-design/charts`)
- **客户区域分布**:按镇街 groupBy 的柱状图,空时给出「暂无区域分布数据」空态
- **客户结构分布**:按规模 / 类型 / 状态三组展示,标签已翻译为中文(企业/政府/其他、大型/中型/小型/微型、线索/洽谈中/已签约/已流失/已冻结)
- **导出**:右上「导出 Excel」,仅 ADMIN / FINANCE 可用(满足 `STATISTICS:EXPORT` 权限)
- **未回款额**:`MAX(0, 已开票额 − 已回款额)`,预付款(invoiceId 为空)不参与挂账时,`未回款额` 自动 clamp 到 0

### 11.2 账龄(Aging)

- **账龄基准切换** (顶部 Segmented):
  - **按到期日** (默认): 以 `Invoice.dueDate` 为起点(合同约定付款日);新发票在开票审核时录入 `dueDate`,历史 ISSUED 发票按 `actualIssueDate + 30 天` 回填,未填写的发票回退到 `actualIssueDate`。
  - **按开票日**: 以 `Invoice.actualIssueDate` 为起点, 与 v0.6 之前行为一致。
- **应收账款账龄分布**:
  - `0-30 天` / `31-60 天` / `61-90 天` / `90+ 天` 四档
  - `actualIssueDate` 在未来(系统时钟漂移/录错)统一归入 `90+`
- **筛选卡片** (顶部 QueryFilter):
  - 账龄段多选(默认全选); 客户 / 业务人员 / 合同(支持搜索) ; 最小金额
  - 重置回到默认(按到期日 + 全选桶 + 全部客户/业务人员/合同)
- **KPI**: 应收总额 / 90+ 余额(占比 + 风险方向) / 最高单笔欠款(链接到发票) / 涉及客户数 / 涉及业务人员数
- **趋势 + 桶图** 同行展示: 左侧 4 桶柱图, 右侧近 30 天应收总额 Line Chart
- **Tab 1: 明细** — ProTable 服务端分页(20/50/100), 列: 发票号 / 客户 / 合同 / 业务人员 / 桶 / 逾期天数 / 剩余未收 / 状态 / 催收 Badge / 操作(添加催收)
  - 表头 Tab label 显示 "明细 (N)" — N 为全部超期数, 跨页时此 N 不变
- **Tab 2: 按客户** — 客户维度 Top 20, 列: 客户 / 发票数 / 总应收 / 4 桶金额 / 90+ 占比(按 20%/50% 染色)
- **Tab 3: 按业务人员** — 合同 owner 维度 Top 20, 列同客户 (SALES 只能看到自己, 其它角色看全员)
- **Tab 4: 未开票合同** — ACTIVE 但还没 ISSUED 发票的合同, signDate 升序, `已过天数 > 30` 标红
- **催收闭环**:
  - 行内"添加催收"按钮(需 `DUNNING:CREATE`) → Drawer 表单: 状态 Segmented / 联系方式 / 最近联系时间 / 备注; PROMISED 必填承诺日
  - 状态机: `已联系 → 客户承诺(必填承诺日) → 已回款(回款到账后可在 /invoices 页操作)`, 或 `争议 / 法务介入`
  - 历史记录在 Drawer 内列表展示, 倒序(最近联系在前)
  - SALES 行级隔离: 只看自己 owner 合同下的催收
- **已退款处理**:已 `refund` 的回款视为从未入账,`remaining` 按发票金额回弹;若该发票随后又有新的 `CONFIRMED` 回款,`remaining` 仍按累计生效回款扣减
- **移动端**:KPI 单列堆叠(2 列);Tab 折叠到 Top 5;明细表 `sticky` 表头;趋势图 / 桶图分两行(每个 `xs={24}`)
- **导出**: Excel `账龄分析_<basis>_<YYYY-MM-DD>.xlsx`, 走 `/api/statistics/export?type=aging` 带全部当前 filter

### 11.3 业绩(Performance)

- **员工业绩**:
  - SALES 只能看自己(短路径 short-circuit,只返回自己一行)
  - ADMIN / FINANCE / OPS / EXPERT 看全员:排除 `isSystem=true` 的内部 actor 与 `role.code = "ADMIN"` 的管理员
  - 顶部支持单人筛选(`?userId=xxx`),后端在 `owners` 上加 `id = userId` 过滤
- 指标:合同金额、开票金额、回款金额、合同数,以及每个员工的「开票率 / 回款率」Tag
- 时间区间可配(`from / to` 同时作用于合同签订日 / 发票开具日 / 回款到账日)
- 移动端同上

### 11.4 Top 客户

- 独立 API(`/api/statistics/top-customers`),通常被总览 / 账龄页引用
- 维度:合同金额(`metric=contract`) / 回款金额(`metric=payment`),Top N(默认 10,上限 50)
- 维度过滤:`metric=contract` 时只列 `total > 0` 的客户;`metric=payment` 时只列 `paymentTotal > 0` 的客户
- 时间区间:`from / to` 同时作用于合同签订日 / 发票开具日 / 回款到账日
- 导出支持 `top-customers` 模式(单次最多 `EXPORT_MAX_ROWS` 行)

## 12. 系统管理

**路径**:`系统管理 / 子模块`
**访问权限**:仅 **管理员(ADMIN)** 和 **指定授权角色**

### 12.1 用户管理

- 列表:工号、姓名、邮箱、电话、角色、部门、状态(ACTIVE / DISABLED)、最后登录时间
- **新建用户**:工号(全大写,唯一)、姓名、邮箱、密码(由创建者设定,**无默认密码**;≥ 8 字符,建议密码管理器生成的随机串)、角色、部门
- **状态切换**:`ACTIVE ↔ DISABLED`,禁用后该用户 **30 秒内** 无法登录 / 自动失效
- **重置密码**:管理员在弹窗里手动输入新密码(8 ~ 72 字符)后确认,旧密码立即失效且清零失败计数/锁定;命令行救火方式仍是 `pnpm reset-password --employeeNo <id> --password <newPwd>`(不传 `--password` 时脚本会生成 20 字符强密码). 重置后自动标记 `mustChangePassword=true`, 用户首次登录会被强制改密
- **角色调整**:改角色后,前端菜单 2s 内重新计算,服务端 2s TTL 缓存 + JWT `roleVersion` 双保险同步失效
- **解锁**:若用户被自动锁定(5 次密码错),管理员在用户列表点「重置密码」即可清掉 `lockedUntil` / `failedLoginCount`. 也可等 15/60 分钟自动解锁

### 12.2 角色管理

- 5 个 **系统角色**(`ADMIN / SALES / FINANCE / OPS / EXPERT`)不可删除,但可改权限矩阵
- 支持 **新建自定义角色** + 配置 **资源 × 操作** 矩阵
- 权限粒度:`{ resource, actions: ["READ", "CREATE", ...] }`
- 修改角色权限后,2s TTL + JWT `roleVersion` 校验, ≤ 2s 内对其他在线用户生效

### 12.3 部门管理

- **树形结构**:`parentId` 自引用,支持任意层级
- 字段:部门编码(唯一,如 `biz` / `tech` / `tech_ops`)、名称、是否启用、排序
- 停用部门后(`isActive=false`),关联用户不再显示该部门选项(下拉过滤)
- **不允许** 把有在职用户的部门硬删除(只能停用)

### 12.4 数据字典

- **类别管理**:每类字典一个 `category`,如 `CUSTOMER_INDUSTRY` / `SERVICE_TYPE` / `CUSTOMER_SOURCE` 等
- **条目管理**:每条 `{ code, label, sort, isActive }`,`code` 在同一类别内唯一
- 字典项 **被业务数据引用时,不允许硬删除**;只能停用
- 调整后 **全局生效**,无需重启

### 12.5 操作日志

- 写入时机:**所有 CUD 操作**(Customer / Contract / Invoice / Payment / User / Role / Department / Dictionary / EmployeeProfile ...)
- 字段:操作人、操作类型(模块)、实体 ID、变更 diff、IP、时间
- 支持按 **操作人 / 模块 / 时间区间** 检索
- 不可手工删除

## 14. 常见问题 FAQ

### Q1. 登录页 7 天自动登录没生效?

检查两件事:
1. **登录前确实勾选了**「7 天内自动登录」复选框
2. 浏览器 **没有开启隐身 / 无痕模式**(无痕模式不持久化 cookie)

### Q2. 提交合同审批被拒,提示「请上传附件」?

合同 `→ ACTIVE` **必须** 至少 1 个附件(参见 [R-04](#62-关键校验系统自动)):
- 回到合同详情 → 编辑附件 → 上传合同扫描件 → 再走「提交审批」
- 附件格式:PDF / Word / Excel / 图片;**单文件 ≤ 20MB**,**单合同 ≤ 5 个附件**

### Q3. 开票提示「金额超过合同总额」?

参见 [R-08](#82-关键校验):
- 同一合同的 **已开 + 当前申请 ≤ 合同总额**
- 解决方案:核对该合同已开票金额 → 调整本次开票金额 / 申请合同变更

### Q4. 回款 confirm 时提示「银行流水号重复」?

参见 [R-10](#92-关键校验):
- `CONFIRMED` 状态流水号全局唯一
- 如果是误录:把状态改回 `PLANNED`,修改流水号后再 confirm

### Q5. 登录提示「账号已被锁定,请稍后再试」?

连续 5 次密码错误账号会被锁 15 分钟, 第 6 次起锁 60 分钟. 处理方式:
- **等自动解锁**: 锁定到期后(`User.lockedUntil` 过期)即可重新登录
- **联系管理员解锁**: 管理员在「用户管理 → 重置密码」即可清锁定并清失败计数

### Q6. 登录页点了「忘记密码」, 重置链接怎么拿到?

v0.10.0 起, 重置流程是 **用户申请 + 管理员送达** 的内部闭环:
1. 登录页右下「忘记密码」→ 输入工号 + 邮箱 → 提交(系统统一返回「已生成」, 不暴露账号是否存在)
2. 管理员在 `OperationLog` 页面查 `action=PASSWORD_RESET_LINK` 的事件, 拿到 `diff.url` 字段的完整重置链接
3. 管理员通过电话 / 现场 / IM 等内部渠道把链接发给用户
4. 用户在 30 分钟内打开链接, 设置新密码(≥ 8 字符), 自动跳回登录页

> 设计取舍: 当前没有邮件 / IM 自动发送通道, 把链接写 OperationLog 让管理员人工送达, 比挂一个永远不发邮件的「假流程」更安全. 等接 IM 后, 把 `app/api/auth/password-reset/request/route.ts` 里 `prisma.operationLog.create` 替换成发送调用即可, 流程不变.
- 如果是真实同号:请联系管理员排查(极小概率)

### Q5. 看不到同事创建的客户?

- 你是 SALES 角色,默认 **行级隔离**:只看到自己作为 `ownerUserId` 的客户
- 如需查看他人客户,联系管理员把你调整到该客户的负责人,或由管理员直接授权

### Q6. 移动端打开页面,按钮特别小?

- 浏览器可能开了「桌面版网站」,请关闭该选项
- 确认浏览器未启用「文字大小」缩放
- 若仍异常,截图发给管理员,带上机型 / 系统 / 浏览器版本

### Q7. 怎么导出数据?

- **列表页右上角「导出」** 按钮(管理员 / 业务 / 财务可用):
  - 客户列表:导出全部或按筛选条件
  - 合同 / 项目 / 发票 / 回款:同上
- **统计页「导出 Excel」**:总览 / 账龄 / Top 客户 / 业绩(SALES 无权限)
- 导出文件格式:`.xlsx`,文件名按「类型 + 时间戳」命名(如 `区域统计_2026-06-28.xlsx`、`合同列表_2026-06-28.xlsx`),中文文件名在所有现代浏览器都能正确显示(走 RFC 5987 编码,服务端 `lib/excel.ts` `attachmentHeader()` 统一处理)

### Q8. 审批中合同能不能改字段?

- 发起人可点「撤回」回到 `DRAFT` 重新编辑
- 撤回会留痕,管理员能看到撤回历史

### Q9. 软删除的数据能恢复吗?

- 列表默认隐藏已软删数据
- 管理员可在筛选中开启「含已删除」,能看到 `deletedAt` 时间戳
- **暂未提供前端恢复入口**,需联系 DBA / 管理员在数据库层恢复(`UPDATE ... SET deletedAt = NULL WHERE id = ?`)

## 15. 附:业务编号与编号规则

系统自动生成的业务编号全部由 `Sequence` 表 + 行锁保证并发安全。

| 对象 | 编号格式 | 示例 |
|---|---|---|
| 客户 | `QT-C-{YYYYMM}-####` | `QT-C-202606-0001` |
| 合同 | `QT-HT-{YYYY}-####` | `QT-HT-2026-0005` |
| 项目 | `QT-P-{YYYY}-####` | `QT-P-2026-0012` |
| 发票(草稿) | `DRAFT-{timestamp}` | `DRAFT-1718188800000` |
| 发票(正式) | `INV-{YYYYMMDD}-{####}` | `INV-20260612-0003` |
| 回款 | `QT-PAY-{YYYY}-####` | `QT-PAY-2026-0008` |

- 月度按 `YYYYMM` 重置(客户);年度按 `YYYY` 递增(合同 / 项目 / 回款)
- 并发安全:Prisma 事务内 `SELECT ... FOR UPDATE` 行锁
- 编号 **不可手工指定**,由系统保证全局递增且唯一

## 16. 运维小贴士(cron 健康与强关预警)

> 本节面向**运维 / 技术接手者**。业务用户可跳过。
> 历史教训:2025-09 ~ 2026-06 期间 cron 静默失败 9 个月无人察觉(详见 `docs/cron-silent-failure-postmortem.md`)。本节是该事故的产物。

### 16.1 快速自检:cron 是否在跑

每次接手巡检时,跑下面三个命令,30 秒内确认 cron 状态:

```bash
# 1) crond 服务本身是否 active
systemctl is-active crond || systemctl is-active cron

# 2) /var/log/qt-cron.log 最后一条记录时间 (2 小时内算正常)
tail -1 /var/log/qt-cron.log
# 或用自检脚本
/opt/qt/scripts/ops/cron-healthcheck.sh --verbose

# 3) 手动触发一次 run-all, 看是否 200
curl -X POST -H "Authorization: Bearer ${CRON_SECRET}" \
  http://127.0.0.1:3000/api/jobs/run-all | head -c 500
```

### 16.2 cron-healthcheck 自检脚本

`scripts/ops/cron-healthcheck.sh` 每小时第 5 分钟跑一次,检查 4 个维度:

- ✅ crond / cron 服务 active
- ✅ `/var/log/qt-cron.log` 最近 2 小时有写入(run-all 跑过的痕迹)
- ✅ qt-app 监听 3000 端口
- ✅ PostgreSQL 容器 healthy

**失败时**:
- 写日志到 `/var/log/qt-cron.log`(运维巡检可见)
- (可选) 推飞书 webhook,需在 `.env` 配置 `FEISHU_WEBHOOK_URL=https://open.feishu.cn/...`

### 16.3 强关前 7/3/1 天预警

合同 `endDate` 过期后,系统每天给 owner + admin 发 `CONTRACT_EXPIRED_UNPAID` 站内信。**强关前 7/3/1 天的消息文案会自动升级**(从 bus.ts 文案模板):

| 剩余天数 | 文案标题 |
|---|---|
| 7 / 3 / 1 | ⚠️ **【强关预警】** 合同 ... — N 天后系统将自动强关 |
| 0 | ⚠️ ... — **今天将被系统强关** |
| 其他 (8+) | 合同 ... 还剩 N 天进入宽限期强关 |

**强关后**:发 `CONTRACT_AUTO_OVERDUE_TERMINATED` 通知(`reason=overdue_terminated`)。

### 16.4 部署后必做(deploy.sh 已自动化)

`scripts/prod/deploy.sh` deploy 完成后会自动跑:

1. ✅ 检查 `crond` / `cron` 服务 active
2. ✅ 检查 `/etc/cron.d/qt-jobs` 包含 `source .env`(防 CRON_SECRET 401)
3. ✅ 立即触发一次 `run-all`,验证 token / API 通畅
4. ✅ 跑一次 `cron-healthcheck.sh`

如果 deploy 报错停在"cron 健康检查"段,**不要强制继续**。先排查:

| 报错 | 排查 |
|---|---|
| `neither crond nor cron is active` | `systemctl start crond`(或 `cron`) |
| `/etc/cron.d/qt-jobs 漏 source .env` | `sudo cp ops/qt-jobs.cron /etc/cron.d/qt-jobs && sudo systemctl restart cron` |
| `run-all 自检 HTTP 401` | 比对 `.env` 和 `/etc/cron.d/qt-jobs` 的 `CRON_SECRET` 是否一致 |
| `cron-healthcheck 有异常` | 看 `/var/log/qt-cron.log` 的 FAIL 行 |

### 16.5 "假完结合同"应急处理

如果发现合同状态 = CLOSED 但实际还有应收未结清(典型场景:cron 长期未跑后批量强关):

1. **批量数据修复** (SQL 脚本, 242 个合同 269 万应收案例可参考):
   ```bash
   pnpm tsx scripts/migrate/contract-fake-close-recovery.ts --execute
   ```
2. **代码层修复** (P2-3 已加):
   - `POST /api/contracts/[id]/reopen` — admin 重开已完结合同
   - `POST /api/payments` body 加 `force: true` + `forceReason: string` — admin 在 CLOSED 合同上强制录回款(自动加 `[FORCE_BACKFILL]` 审计标记)

3. **完整流程文档**:`docs/contract-fake-close-recovery.md`
